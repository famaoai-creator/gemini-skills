# リリースノート記述のベストプラクティス (SS/JSM基準)

## 1. 構成セクション
リリースノートは「誰が読むか」によって以下のセクションを設ける。

### A. エグゼクティブサマリー (経営層・PM向け)
- **リリース目的**: なぜこのリリースを行うのか（法改正対応、障害対応、機能改善など）。
- **ビジネス価値**: このリリースによってユーザーや業務にどのようなプラスの影響があるか。
- **影響範囲**: リリースに伴うダウンタイムや、ユーザーへの通知事項。

### B. 主要な変更点 (ステークホルダー向け)
- **Jira連携**: チケットのタイトル (Summary) をそのまま使わず、一言でメリットが伝わる表現に変換する。
- **例**: `IB-24311 アニメーション発火修正` → `目的別口座画面の表示速度およびUXの改善`

### C. 修正・メンテナンス (QA・CS向け)
- **バグ修正**: どのような事象が解消されたか。
- **セキュリティ・コンプライアンス**: ライブラリの更新や脆弱性対応、FISC基準への準拠状況。

### D. 技術的詳細 (エンジニア・SRE向け)
- **GitHub PR**: 関連するプルリクエストのURL。
- **バージョンタグ**: GitHub上の新旧タグ（例: `v1.2.3` -> `v1.3.0`）。
- **デプロイ資材**: 影響を受けるコンポーネント（API, Web, DB, IaC等）。

### E. 作業・復旧計画 (オペレーション向け)
- **Confluenceリンク**: 「リリース手順書」および「作業ランブック」への直接リンク。
- **ロールバック方針**: 有事の際の切り戻し手順の有無と、その判断基準。

## 2. 記述のルール
- **証跡性**: Jiraチケット、GitHub PR、Confluenceページが相互にリンクされていること。
- **平易な表現**: 技術用語のみに偏らず、ビジネスインパクトを明記する。
- **一貫性**: リポジトリ名やバージョン表記はGitHub上の実体と一致させる。
