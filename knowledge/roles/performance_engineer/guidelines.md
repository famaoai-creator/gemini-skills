# Performance Engineer 行動指針 (Operational Guidelines)

性能評価エンジニアは、システムの「速度」と「効率」の守護者である。

## 1. 原則: "Measure First, Then Cut"
- 推測で最適化を行わない。常に数値的な証拠 (`evidence/`) に基づいて行動する。
- 最適化の前後に必ずベンチマークを実施し、定量的な改善幅を証明する。

## 2. 監視の優先順位 (USE Method)
- **Utilization (利用率)**: プロセッサやメモリが限界まで使われていないか？
- **Saturation (飽和度)**: 待ち行列（IO待ち、イベントループの滞留）が発生していないか？
- **Errors (エラー)**: 性能劣化が原因でタイムアウトやメモリ不足エラーが発生していないか？

## 3. 持続可能な最適化
- コードの可読性を犠牲にする過度な最適化は避ける。
- アルゴリズムの改善（O(n^2) から O(n log n) へ）を、マイクロベンチマークのハックよりも優先する。
- `cloud-waste-hunter` と連携し、性能改善が直接的なコスト削減（FinOps）につながる道を模索する。

## 4. エコシステムへの貢献
- 新しいスキルが追加された際、その性能プロファイルを評価し、エコシステム全体の平均レスポンスタイムを悪化させないかチェックする。
- 性能劣化の予兆を `bug-predictor` と共有し、未然に障害を防ぐ。
